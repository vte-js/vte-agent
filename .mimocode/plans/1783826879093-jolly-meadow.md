# 多 Agent 并行系统设计与实现

## 目标

实现多 Agent 并行执行能力，支持工单池调度、管道式、并行分工和混合模式。每个 Agent 独立 LLM 配置，可配置共享或隔离的执行环境。

## 当前架构分析

### 单 Agent 架构
```
panel.ts → AgentEngine (单实例)
  ├── context: ContextManager (文件操作)
  ├── messages: AgentMessage[] (对话历史)
  ├── tools: ToolDefinition[] (全局工具注册)
  └── onViewUpdate → webview
```

### 关键限制
- `AgentEngine` 是单实例，管理一个对话循环
- `panel.ts` 持有单个 `this.engine`
- 任务是全局模块级数组（`tasks.ts`）
- 工具注册是全局的（`registry.ts`）

### 2. 工单系统

```typescript
// src/agent/work-order.ts

export interface WorkOrder {
  id: string
  title: string
  description: string
  status: 'pending' | 'assigned' | 'running' | 'done' | 'failed' | 'blocked'
  assignedRole?: string         // 分配给哪个角色
  assignedAgentId?: string      // 分配给哪个具体 Agent
  parentId?: string             // 父工单
  dependencies: string[]        // 依赖的工单 ID
  result?: string               // 执行结果
  error?: string                // 错误信息
  createdAt: string
  updatedAt: string
  startedAt?: string
  completedAt?: string
  metadata?: Record<string, unknown>
}

export interface WorkOrderPool {
  orders: Map<string, WorkOrder>
  add(order: Omit<WorkOrder, 'id' | 'createdAt' | 'updatedAt'>): WorkOrder
  get(id: string): WorkOrder | undefined
  update(id: string, updates: Partial<WorkOrder>): WorkOrder | undefined
  complete(id: string, result: string): void
  fail(id: string, error: string): void
  getPending(): WorkOrder[]
  getByRole(role: string): WorkOrder[]
  getReady(): WorkOrder[]  // 依赖已满足的待处理工单
}
```

### 3. Agent 池管理器

```typescript
// src/agent/agent-pool.ts

export interface AgentInstance {
  id: string
  role: AgentRole
  engine: AgentEngine
  status: 'idle' | 'busy' | 'error'
  currentOrderId?: string
  completedOrders: number
  failedOrders: number
}

export class AgentPool {
  private agents: Map<string, AgentInstance> = new Map()
  private workOrderPool: WorkOrderPool
  private scheduler: Scheduler

  constructor(host: HostAdapter) { ... }

  // 创建 Agent 实例
  createAgent(role: AgentRole): AgentInstance

  // 获取空闲 Agent
  getIdleAgent(role?: string): AgentInstance | undefined

  // 分配工单给 Agent
  assignOrder(orderId: string, agentId: string): void

  // 启动 Agent 执行工单
  async executeOrder(agentId: string, orderId: string): Promise<void>

  // 获取所有 Agent 状态
  getAgentStatuses(): AgentStatus[]

  // 停止所有 Agent
  stopAll(): void
}
```

### 4. 调度器

```typescript
// src/agent/scheduler.ts

export type ScheduleMode = 'pool' | 'pipeline' | 'parallel' | 'hybrid'

export interface ScheduleConfig {
  mode: ScheduleMode
  maxConcurrent: number          // 全局最大并发
  autoAssign: boolean            // 自动分配工单
  pipelineStages?: string[]      // 管道模式的阶段顺序
}

export class Scheduler {
  private pool: AgentPool
  private config: ScheduleConfig

  // 自动分配待处理工单
  async autoSchedule(): Promise<void>

  // 管道模式：按阶段串行执行
  async runPipeline(orders: WorkOrder[]): Promise<void>

  // 并行模式：同时执行所有工单
  async runParallel(orders: WorkOrder[]): Promise<void>

  // 混合模式：依赖图调度
  async runHybrid(orders: WorkOrder[]): Promise<void>

  // 监听工单完成事件，触发下一步
  onOrderComplete(orderId: string, callback: (result: string) => void): void
}
```

### 5. Agent 间通信

```typescript
// src/agent/agent-communication.ts

export interface AgentMessage {
  id: string
  from: string        // 发送者 Agent ID
  to?: string         // 接收者 Agent ID（空=广播）
  type: 'task_result' | 'request_help' | 'share_context' | 'notification'
  content: string
  metadata?: Record<string, unknown>
  timestamp: string
}

export class AgentCommunication {
  private handlers: Map<string, (msg: AgentMessage) => void> = new Map()

  // 发送消息
  send(from: string, to: string | undefined, msg: Omit<AgentMessage, 'id' | 'timestamp'>): void

  // 监听消息
  on(agentId: string, handler: (msg: AgentMessage) => void): void

  // 广播
  broadcast(from: string, msg: Omit<AgentMessage, 'id' | 'timestamp'>): void
}
```

## 已知问题

### AgentConversation 实时显示（待修复）
**问题**：`executeOrder()` 运行多轮 LLM 循环，但 `conversationHistory` 只在整个循环结束后更新。执行期间只显示初始任务描述。

**原因**：
- `executeOrder()` 内部调用 `engine.chat()` 进入多轮循环（callLLM → 工具执行 → callLLM）
- `llmResponseText` 在 `onViewUpdate` 中累积所有 stream_chunk 文本，但只在整个循环结束后才存入 `conversationHistory`
- 工具调用事件（tool_call/tool_result）在执行过程中已经通过 `onAgentUpdate` 发送到 webview，但 AgentConversation 不读取这些事件

**修复方向**：
- 方案 A：在 AgentInstance 中维护一个实时事件流（events 数组），`onAgentUpdate` 写入，AgentConversation 订阅
- 方案 B：在 `executeOrder()` 中每轮 LLM 完成后（finish_reason != tool_calls）立即更新 conversationHistory
- 推荐方案 A：更干净，不依赖循环结构

### LLM 执行效率
- LLM 在没有明确任务描述时会反复探索（如工单标题为"新任务"）
- 已通过移除 ENFORCEMENT 和软化 prompt 缓解
- 根本原因：LLM 模型本身特性，需通过更明确的工单描述改善
